## 技术选型决策

### 后端: Go + Gin + Ent

| 维度 | 选型 | 理由 |
|------|------|------|
| 语言 | Go | 高并发低内存，适合 Tick 引擎 (每5分钟全量农场状态处理)；编译为单一二进制文件，部署简单 |
| HTTP 框架 | Gin | 高性能、社区成熟、中间件生态丰富 |
| ORM | Ent (Meta) | 代码生成型类型安全 ORM，编译时检查无运行时反射；内置 Migration 版本管理；Schema-as-Code |
| WebSocket | Gorilla WebSocket | Go 生态标准 WebSocket 库 |
| 定时任务 | robfig/cron | 轻量 cron 调度器，用于 Tick 引擎和日常调度 |
| API 文档 | swaggo/swag | 从注释生成 OpenAPI spec，实现前后端类型共享 |
| 缓存 | Redis | 热数据层: 会话/在线状态/排行榜(Sorted Set)/Tick 缓存/限流 |

### 前端: Taro 3 (React) vs UniApp 对比

| 维度 | Taro 3 (React) | UniApp (Vue) | 决策 |
|------|---------------|-------------|------|
| PixiJS 集成 | `@pixi/react` 成熟方案 | 需手动桥接 | Taro 胜 |
| 复杂状态管理 | React hooks + Zustand，适合游戏级嵌套状态 | Pinia，嵌套状态略弱 | Taro 胜 |
| Canvas 生命周期 | ref + useEffect 精确控制 | 需 onReady + $nextTick | Taro 胜 |
| 双端编译 | 原生支持小程序 + H5 | 原生支持小程序 + H5 | 平手 |
| 生态 | React 生态更大 | Vue 中文生态更大 | 视团队 |

**决策: Taro 3 (React)**  —— PixiJS 集成和游戏状态管理是关键差异点。

### 前后端类型共享方案

```
Go 代码 (swaggo 注释)
    ↓ swag init
OpenAPI spec (swagger.json)
    ↓ openapi-typescript-codegen
TypeScript 类型 + API Client
    ↓ import
Taro 前端直接使用
```

### 存储架构

```
┌─ PostgreSQL (Ent ORM) ─── 持久化层，唯一真相源
│   全部核心数据 + JSONB 存储农场格子等灵活结构
│
├─ Redis ─── 热数据层
│   会话/在线状态/排行榜(Sorted Set)/Tick 缓存
│   LLM 对话缓存/限流/Pub-Sub 广播
│
├─ 客户端 Taro.setStorage ─── 本地缓存
│   Token、用户设置、资源缓存
│
└─ OSS/S3 ─── 静态资源
    像素 Sprite Sheet、音效、CDN 加速
```

**农场状态读写流程**:
1. 上线: PG → Redis (加载热数据)
2. 操作: 写 Redis → 异步 batch 写 PG
3. 下线: flush Redis → PG

**消息队列策略**: MVP 阶段不引入独立 MQ，用 Go goroutines/channels 处理异步任务 + Redis Pub/Sub 处理广播。5000+ 并发后考虑加 NATS/RabbitMQ。

## Phase 1: MVP 开发 — P0 功能 (10 周, 5 个 Sprint)

> 目标: 实现 PRD 9.1 P0 功能范围，可发布微信小程序灰度版。
> 核心验证: 种植循环可玩性 + AI Agent 自动经营 + 村庄社交。

### Sprint 1: 用户系统 + 种植核心 + 前端框架 (Week 3-4)

- [x] **T-011** `[M]` **用户注册与微信登录**
  - Ent Schema: User (id, openid, nickname, avatar, level, exp, coins, diamonds, stamina)
  - 微信小程序 `wx.login` → 后端 code2session → JWT 签发
  - 中间件: JWT 认证 + 用户上下文注入
  - API: `POST /api/auth/wx-login`, `GET /api/user/profile`

- [x] **T-012** `[S]` **用户会话与 Redis 在线状态**
  - Redis 存储: 用户会话 (token → userId)、在线状态 (SET)
  - 上线/下线事件记录 (用于离线回放)
  - 体力值自动恢复逻辑 (每 6 分钟 +1, 上限 100)

- [x] **T-013** `[L]` **农场数据层 (Ent Schema)**
  - Farm Schema: id, owner_id, name, level, grid (JSONB 8×8)
  - Plot 结构 (嵌入 JSONB): type, crop_id, planted_at, stage, quality, watered
  - Building Schema: id, farm_id, type, level, position, state
  - Crop 配置表 (静态数据): 14 种作物参数 (对应 PRD 4.1)
  - 新用户自动创建初始农场 (4 块农田 + 1 仓库)

- [x] **T-014** `[L]` **种植核心逻辑 — 播种与生长**
  - API: `POST /api/farm/plant` (地块id, 种子id) → 扣种子、设置生长开始时间
  - 生长阶段计算: 基于 planted_at + crop.growTime 计算当前阶段 (seed → sprout → mature → harvestable)
  - 解锁条件校验: 玩家等级 ≥ 作物要求等级
  - 体力值扣除 (手动操作消耗体力)

- [x] **T-015** `[M]` **种植核心逻辑 — 收获与仓库**
  - API: `POST /api/farm/harvest` (地块id) → 判断成熟 → 产出入库 → 重置地块
  - Inventory Schema: id, user_id, item_id, item_type, quantity
  - 仓库容量检查 (Lv.1 = 100 格)
  - 收获结果: 产出数量 ×1, 金币奖励, 经验值

- [x] **T-016** `[L]` **Tick 引擎 — 全局农场状态处理**
  - robfig/cron 定时任务: 每 5 分钟触发一次 Tick
  - Tick 流程: 遍历所有在线农场 → 更新作物生长阶段 → 检查成熟 → 触发 AI 行为
  - Redis 批量读写: 从 Redis 读取热数据 → 处理 → 写回 Redis
  - 异步 PG 回写: goroutine 批量 flush 到 PostgreSQL (每 3 次 Tick)
  - 性能目标: 1000 农场 / Tick < 2 秒

- [x] **T-017** `[L]` **前端 — Taro 页面骨架与路由**
  - 页面结构: 首页(农场) / 村庄 / 集市 / 背包 / 个人中心
  - TabBar 导航配置
  - Zustand store 结构: userStore, farmStore, villageStore, uiStore
  - API service 层: 基于 OpenAPI 生成的 TypeScript client
  - 全局 loading/error 状态处理

- [x] **T-018** `[L]` **前端 — PixiJS 农场画布组件**
  - `<FarmCanvas>` 组件: PixiJS Application 生命周期管理
  - 8×8 网格渲染 + 地块 sprite 绑定 (空地/种植中/可收获)
  - 点击交互: 选中地块 → 底部弹出操作面板 (种植/收获/浇水)
  - 农场数据绑定: farmStore → Canvas 重绘
  - 适配小程序 Canvas 2D (Taro.createCanvasContext)

- [x] **T-019** `[S]` **前端 — 种植操作流程 UI**
  - 种子选择弹窗: 展示可用种子列表 (图标+名称+生长时间+解锁状态)
  - 种植确认 → 调用 API → 地块动画更新
  - 收获操作 → 飘字动画 (+金币 +经验)
  - 生长进度条 (显示在地块上方)

- [x] **T-020** `[M]` **API 中间件与错误处理**
  - 统一响应格式: `{code, message, data}`
  - 错误码体系: 业务错误 (40xxx) / 系统错误 (50xxx)
  - 请求日志中间件 (structured logging with slog)
  - 限流中间件: Redis 令牌桶，API 级别限流

- [x] **T-021** `[M]` **CI/CD Pipeline 搭建**
  - GitHub Actions: lint → test → build → Docker image
  - Go: golangci-lint + go test + go build
  - Taro: eslint + tsc --noEmit + taro build
  - Docker 镜像推送到 registry
  - 开发/测试环境自动部署

- [x] **T-022** `[S]` **数据库 Migration 工作流**
  - Ent auto-migration 配置 (开发环境)
  - Atlas versioned migration (生产环境)
  - 种子数据脚本: 作物配置表、建筑配置表初始数据
  - `make migrate-up / migrate-down` 命令

### Sprint 2: 仓库系统 + 建筑 + AI Agent 规则引擎 (Week 5-6)

- [x] **T-023** `[L]` **仓库系统 — 物品管理**
  - API: `GET /api/inventory` (分页+分类筛选)
  - API: `POST /api/inventory/use` (使用物品, 如种子)
  - API: `POST /api/inventory/sell` (出售物品, 获得金币)
  - 物品分类: 种子/作物/动物产品/建材/特殊道具
  - 前端: 背包页面 — 网格展示、拖拽排序、批量出售

- [x] **T-024** `[L]` **建筑系统 — 建造与升级**
  - Building 配置表: 9 种建筑 × 3 级参数 (对应 PRD 4.3)
  - API: `POST /api/farm/build` (建筑类型, 位置) → 扣资源 → 创建建筑
  - API: `POST /api/farm/upgrade` (建筑id) → 校验升级条件 → 扣资源 → 升级
  - 建筑效果激活: 水井自动浇水、仓库扩容
  - 前端: 建造面板 UI (建筑列表+费用+位置选择)

- [x] **T-025** `[L]` **AI Agent 规则引擎 — 决策树**
  - Agent 数据结构: personality (3轴), strategy_prefs (5项滑块), state
  - 决策树实现: 空地 → 选种策略 → 播种; 成熟 → 收获; 满仓 → 出售
  - 策略偏好影响: 保守/激进 → 短周期/长周期作物偏好
  - 在 Tick 引擎中调用: 每个 Tick 为每个在线农场执行一次 Agent 决策
  - 行为日志: 记录 Agent 每次操作 (用于动态流展示)

- [x] **T-026** `[M]` **AI Agent 自动种植与收获**
  - 自动种植: 检测空地 → 根据策略选择作物 → 执行种植
  - 自动收获: 检测成熟地块 → 执行收获 → 入库
  - 自动浇水: 水井建筑等级决定频率 (Lv.1手动/Lv.2半自动/Lv.3全自动)
  - 操作通知: Agent 行为生成简报 (玩家上线后查看)

- [x] **T-027** `[M]` **策略偏好面板 — 前端**
  - 5 项滑块 UI 组件 (对应 PRD 3.4)
  - 经营风格 / 种植偏好 / 社交倾向 / 交易策略 / 资源分配
  - 每项 5 级刻度，保存到后端
  - API: `GET /PUT /api/agent/strategy`
  - 实时预览: 修改偏好后展示 Agent 行为倾向说明

- [x] **T-028** `[M]` **人格系统初始化**
  - 新玩家注册时随机生成 3 轴人格值 (外向/慷慨/冒险, 各 1-10)
  - 人格影响 Agent 行为参数 (如外向 → 社交频率倍率)
  - API: `GET /api/agent/personality`
  - 前端: Agent 个人页 — 展示人格雷达图

- [x] **T-029** `[S]` **经验与等级系统**
  - 经验值获取: 种植 +5, 收获 +10, 交易 +8, 建造 +20
  - 等级公式: level = floor(sqrt(exp / 100))
  - 升级触发: 解锁新作物/建筑, 推送通知
  - 前端: 经验条 UI (顶部HUD)

- [x] **T-030** `[S]` **浇水与体力系统**
  - 浇水机制: 手动浇水消耗 2 体力, 加速生长 20%
  - 体力系统: 上限 100, 每 6 分钟恢复 1 点
  - API: `POST /api/farm/water` (地块id)
  - 前端: 体力条 UI + 浇水按钮 + 浇水动画

### Sprint 3: 村庄系统 + 社交行为 + 实时通信 (Week 7-8)

- [x] **T-031** `[L]` **村庄数据层与匹配**
  - Village Schema: id, name, level, contribution, member_count, specialty
  - VillageMember Schema: village_id, user_id, role, joined_at, contribution
  - 新玩家自动匹配: 找未满村庄 (10-20人), 优先匹配同时段注册玩家
  - 村庄特产机制: 每个村庄随机分配 1-2 种特产作物 (+30%产量加成)
  - API: `GET /api/village/info`, `GET /api/village/members`

- [x] **T-032** `[L]` **社交关系数据层**
  - Relationship Schema: agent_a, agent_b, affinity (0-100), level (枚举5级), last_interact
  - 好感度变化规则 (对应 PRD 5.3): 串门+5, 闲聊+3, 赠礼+10~20, 交易+5
  - 每日自然衰减 -1, 连续3天无互动 -2/天
  - 关系层级自动升降: 基于好感度阈值 (0/20/50/80/100)
  - API: `GET /api/social/relationships`

- [x] **T-033** `[M]` **Agent 社交行为 — 串门与闲聊**
  - 串门触发: Tick 中按社交倾向概率触发 (外向 Agent 概率更高)
  - 串门流程: 选择目标 (好感度≥20) → 拜访 → 观察农场 → 闲聊
  - 模板对话系统: 50+ 预设对话模板, 根据场景/关系/人格选择
  - 串门结果: 好感度 +5(串门) +3(闲聊), 生成动态日志
  - 频率限制: 每个 Agent 每日最多 3 次串门

- [x] **T-034** `[M]` **模板对话引擎**
  - 对话模板 YAML 配置: 场景(串门/交易/求助) × 关系等级 × 人格倾向
  - 变量替换: {agent_name}, {target_name}, {crop_name}, {season} 等
  - 多轮对话: 每次 2-4 轮, 支持条件分支
  - 降级方案: 确保无 LLM 时也有丰富的社交对话
  - 对话存储: ChatLog Schema (sender, receiver, content, scene, created_at)

- [x] **T-035** `[L]` **WebSocket 服务 — 村庄实时通信**
  - 连接建立: 用户上线 → 加入村庄房间
  - 消息类型: 农场状态更新 / 社交事件 / 交易通知 / 系统公告
  - 房间管理: 按村庄分组, 支持广播和定向推送
  - Redis Pub/Sub: 跨进程消息分发 (为多实例部署准备)
  - 前端 WebSocket 客户端: 自动重连 + 消息队列 + 心跳

- [x] **T-036** `[M]` **村内交易系统**
  - TradeOrder Schema: seller_id, buyer_id, item_id, quantity, price, status
  - API: `POST /api/trade/list` (上架), `POST /api/trade/buy` (购买), `GET /api/trade/orders`
  - 村内交易免手续费
  - 购买成功: 扣金币 → 转移物品 → 双方好感度 +5
  - WebSocket 推送: 新商品上架通知

- [x] **T-037** `[M]` **前端 — 村庄页面**
  - 村庄信息展示: 名称、等级、成员列表、特产
  - 村民列表: 头像 + 名称 + 在线状态 + 好感度
  - 拜访按钮: 点击跳转到目标农场 (只读模式)
  - 村庄动态流: 时间线展示 Agent 社交行为日志

  - 游戏内 1 季 = 现实 1 周, 春→夏→秋→冬 循环
  - 当季加成作物: 每季 2-3 种作物产量 +50%
  - 冬季限制: 部分作物无法种植
  - 季节交替事件: 春耕节/丰收祭 触发
  - 前端: 季节氛围 (背景色/天气动效/当季作物标记)

- [x] **T-083** `[L]` **随机事件系统**
  - 事件引擎: 每日按概率触发随机事件 (对应 PRD 5.5)
  - 7 类随机事件: 流浪商人/暴风雨/动物走失/神秘种子/害虫/彩虹/Agent吵架
  - 事件处理: 部分需玩家响应 (如暴风雨抢救作物), 部分自动结算
  - 社交联动: 动物走失→邻居帮找, 害虫→借农药, 吵架→调解
  - 前端: 事件弹窗 + 事件日志

- [x] **T-084** `[M]` **作物品质系统**
  - 品质分 3 级: 普通/优良/极品 (对应 PRD 4.1)
  - 影响因素: 浇水及时性(±20%) + 施肥(+15%) + 天气(±10%) + 邻居祝福(+10%)
  - 极品作物: 售价翻倍, 可作为赠礼素材
  - 品质动画: 收获时品质揭晓 (类似开箱)
  - 品质统计: 赛季品质产出排名

### Sprint 12: 节日活动框架 + 微信裂变 (Week 25-26)

- [x] **T-085** `[L]` **节日活动框架**

  - Event Schema: id, type, config(JSONB), start_at, end_at, status

- [x] **T-086** `[L]` **微信裂变与分享**

  - 邀请系统 (对应 PRD 10.3): 邀请1/3/5/10人阶梯奖励

## 风险与缓解策略

| # | 风险 | 影响 | 缓解方案 | 关联任务 |
|---|------|------|----------|----------|
| 1 | LLM 调用成本过高 | 运营成本超预算 | 分级调用 + 模板缓存 + 频率限制 + 离线批量生成 | T-058, T-059 |
| 2 | PixiJS 小程序兼容性 | 渲染效果打折或性能不达标 | 小程序 Canvas 2D 降级方案, 早期 POC 验证 | T-005, T-006 |
| 3 | AI 生成内容安全 | 不当内容影响审核和用户体验 | 输出过滤 + 模板兜底 + 举报机制 | T-058, T-052 |
| 4 | 冷启动社交不足 | 新村庄缺乏社交氛围,留存差 | AI 机器人填充 (5-8个/村) + 快速匹配 | T-031, T-043 |
| 5 | Tick 引擎性能瓶颈 | 用户量增长后延迟增加 | 并行处理 + 跳过不活跃农场 + 水平扩展 | T-016, T-094 |
| 6 | 微信小程序审核风险 | 发布延迟 | 提前准备审核材料, 预留审核周期, 合规自查 | T-053 |
